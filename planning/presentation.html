🛠️ System Overview

🧩 Core Components

1. Data Pipeline

Automated ingestion from TMT source.

  • Download: Fetch TMT Excel/JSON
  • Parasing: Clean & Normalize data
  • Embedding: Generate vectors (bge-m3)
  • Graph Insert: Nodes (VTM, GP) & Rels
graph LR A[Raw Data] --> B[Clean] --> C[Vectorize] --> D[(Neo4j)] style D fill:#dcfce7,stroke:#166534

2. RAG System

Hybrid Search Architecture.

  • Query: Classify & Extract Keywords
  • Retrieval: Vector + Fulltext + RRF
  • Expand: Graph Traversal (2-hop)
  • Gen: Ollama (Thai Structured Output)

🏗️ Knowledge Domain Scope

Defining what our current GraphRAG system can and cannot answer.

In-Scope (Data Available)

  • Drug Mapping: "Which brand matches this generic?"
  • Ingredients: "What is the composition of Drug X?"
  • Manufacturers: "Which company produces Y?"
  • Hierarchy: "What are the parent/child nodes of TMT ID Z?"
  • Dose Forms: "Does X come in Syrup or Tablets?"

Out-of-Scope (Missing Data)

  • Clinical Advice: "How to treat disease A?"
  • Side Effects: "Will X make me dizzy?"
  • Dosage Rules: "How many pills per day?"
  • Interactions: "Can I take X with Alcohol?"
  • Logistics: "What is the price or stock status?"

Bottleneck Analysis

  • Serial Execution:
    Search -> Traversal -> Generation happen one by one. LangChain Async can parallelize this.
  • Over-fetching:
    We fetch 13,060 chars of context for a simple question. Agents can "Self-Select" cleaner context.

Why shifting to LangChain isn't just about features, it's about Standardization.

❌ Pain Points: No standard interface, manual error handling, hard to swap components.

The "Standard" Way (LangChain)
# 1. Standard Components
prompt = ChatPromptTemplate(...)
model = ChatOllama(model="qwen2.5")

# 2. Chain Syntax (LCEL)
chain = prompt | model | StrOutputParser()

# 3. Validated Execution
return chain.invoke({"context": vals})

✅ Benefits: Type safety, interchangeable models (Ollama/OpenAI), built-in streaming.
